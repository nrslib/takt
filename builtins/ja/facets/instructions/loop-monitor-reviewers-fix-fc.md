レビュー → 修正のループが {cycle_count} 回繰り返されました。

Finding Contract の台帳サマリー / `findings-ledger.json` が利用できる場合は、統合台帳を主要な根拠にしてください。
台帳がない場合は、Report Directory 内の最新レビュー報告を調べ、このループが健全（収束中）か、非生産的（発散・振動・停滞）かを判断してください。

**最初に観測時点を確定してください:**
- このモニターは、修正完了直後で修正後レビューがまだない時点にも発火します。直近のレビューより後に当該 finding を修正したとの報告があり、その後のレビュー証拠がない場合は「修正後の検証待ち」であり、反復や停滞ではありません。選択肢では健全・進捗ありの条件に含めてください。ただし、後述の provisional 停滞は例外です。
- 台帳で finding が open のままであることだけを、修正の失敗や台帳更新の遅延と判断しないでください。関連する修正より後のレビュー証拠が同じ finding を再確認した場合に限り、persistent / reopened と判断してください。

**判断基準:**
- 解析可能な Finding Contract 台帳がある場合、台帳の `findings` / `conflicts` を正とし、個別レポートは補助証拠としてください。
- 台帳が不完全な場合、対応付け済み finding は台帳に従い、未対応の生の指摘は findings-manager の調停待ちの候補として扱ってください。
- 台帳がない、読めない、または解析できない場合は、Report Directory 内の最新レビュー報告を主要な根拠にしてください。
- 同一 finding_id が複数の修正後レビューでも継続しているか確認してください。
  - 修正後レビュー待ち → 健全（通常の検証を続ける）
  - 修正後レビューが同一 finding_id を再確認 → 非生産的な反復の候補
  - 以前の finding が解消され、別の finding が new で出現 → 原則として健全
  - ただし、直近に解消した finding と同じ `family_tag` の別分岐が new で現れた時点で部分修正の再発として非生産的側に数えてください。同じ family が分岐単位で続く場合は、全分岐を一括修正する再計画が有効か判断してください。
- 修正後レビューでも指摘が現行コードと乖離している場合に限り、異議申告（dispute → manager 裁定 → waive）を打開手段として検討してください。修正後レビュー前の open finding を理由に異議申告や再計画を選ばないでください。
- new / reopened finding の数が全体として減少しているか確認してください。

修正、再計画、異議申告のいずれでも打開できない場合に限り、停止を選んでください。

**engine が注入した「Findings state」セクションがある場合は必ず確認してください:**
- 完了条件（findings.open.count == 0）を妨げる open provisional finding と、その停滞 manager round 数・解消経路（後続の clean evidence / manager の dismissDecisions）が示されます。
- dismiss 可能な provisional や、後続の clean evidence でしか解消できない provisional が複数ラウンド停滞している場合は、解消済み件数が増えていても非生産的と判断し、健全な継続を選ばず、利用可能な裁定または停止の経路を選んでください。
