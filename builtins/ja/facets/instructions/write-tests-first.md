計画に基づいて、プロダクションコードの実装前にテストを作成してください。
Workflow Contextに示されたReport Directory内のファイルのみ参照してください。他のレポートディレクトリは検索/参照しないでください。

**重要: プロダクションコードは作成・変更しないでください。テストファイルのみ作成可能です。**

**やること:**
1. 計画レポートを確認し、実装予定の振る舞いとインターフェースを理解する
2. 計画内の要件を、観測可能な契約に分解する
   - 要件IDがない場合は、このレポート内だけで使う安定IDを付ける
   - 戻り値、保存形式、設定キー、CLI出力、イベント、ログ、エラー分類、副作用を契約として扱う
3. 各契約について、入口/経路を洗い出す
   - CLI、設定読み込み、設定保存、実行時解決、バッチ処理、子処理、イベント生成、永続化境界を確認する
   - 共通ヘルパー、正規化関数、builder、adapter を変更する場合は、既存の同種分岐も入口に含める
4. 対象モジュールの既存コードと既存テストを確認し、テストパターンを把握する
5. 要件ごとのテスト対応表を作り、未カバー項目には理由を残す
6. 計画された機能に対する単体テストを作成する
7. インテグレーションテストの要否を判断し、必要なら作成する
   - 3つ以上のモジュールを横断するデータフローがあるか？
   - 新しいステータス／状態が既存ワークフローに合流するか？
   - 新しいオプションが呼び出しチェーンを通じて末端まで伝搬するか？
   - 保存した値を再読み込みする roundtrip が契約になるか？
   - いずれかに該当すればインテグレーションテストを作成する
8. 作成したテストが、誤実装でも通る弱いテストになっていないか確認する

**テスト作成の方針:**
- プロジェクトの既存テストパターン（命名規約、ディレクトリ構成、ヘルパー）に従う
- Given-When-Then 構造で記述する
- 1テスト1概念。複数の関心事を1テストに混ぜない
- 正常系・異常系・境界値・エッジケースを網羅する
- 要件ごとに、少なくとも1つのテストまたは明示的な未カバー理由を残す
- 外部契約がある場合は、契約どおりの入力位置を使ったテストを含める
  - 例: request body は定義された root shape のまま渡す
  - 例: query / path parameter は body に流用せず、本来の位置から渡す
- レスポンス標準の envelope をリクエスト解釈へ流用した実装を見逃さないテストを含める
- 共通ヘルパー、正規化関数、builder、adapter が契約を担う場合は、既存の同種分岐でも戻り値・副作用・イベント・エラー分類が保たれることを検証する
- 欠落、未知値、不正値、優先順位衝突、override、保存/読み込み roundtrip、部分失敗、外部送信禁止など、実装が落としやすい分岐を優先して固定する
- 「含まれない」だけを確認する absence-only テストに偏らず、観測可能な振る舞いで否定契約を検証する
- テストダブルを使う場合は、対象テストが依存する本番契約を省略しない
- テストは実装完了後にパスすることを前提に書く（この段階ではビルドエラーやテスト失敗は想定内）

**完了条件:**
- 要件-テスト対応表があり、各要件にテストまたは未カバー理由がある
- 代表的な危険分岐、否定契約、横断経路がテストまたは未カバー理由として記録されている
- Large 変更では、単体テストだけでなく主要な入口/経路を通す統合テストがある
- 共有契約を担う helper/normalizer/builder/adapter を触る場合、既存同種分岐の保持テストがある
- 未カバー理由が「時間不足」「未確認」だけで終わっていない

**非実行資産に関する制約:**
- 説明文、ガイド、README、Markdown ドキュメントなどの本文・章タイトル・構成を固定するテストは作成しない
- docs-only 変更では、明示的な実行契約がない限りテストを追加しない
- テストが必要なのは、CLI例・設定例・生成物など、コードの挙動や機械処理と結びつく契約がある場合に限る

**テスト実行について:**
- テスト作成後、テストを実行して結果を確認する
- 実装前のためテスト失敗・import エラーは想定内（未実装モジュールの import を含む）
- 既存モジュールの import パスミスなど、実装完了後も解消しないエラーは修正する
