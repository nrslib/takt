# QA 検出基準

## エラーハンドリングとログ

| 基準 | 判定 |
|------|------|
| エラーの握りつぶし（空のcatch） | REJECT |
| ユーザー向けエラーメッセージが不明確 | 修正が必要 |
| システム境界でのバリデーション欠如 | 警告 |
| 新しいコードパスにデバッグログがない | 警告 |
| ログへの機密情報の出力 | REJECT |

## 観測可能な副作用

メトリクス、ログ、監査記録、セッション記録、イベント、エラー分類などの観測可能な副作用は、実行結果を読む人や後続処理が依存する契約として扱う。

| 基準 | 判定 |
|------|------|
| 対象外の事象で記録・分類・計上される | REJECT |
| 対象の事象で記録・分類・計上されない | REJECT |
| 同じ意味の事象が経路によって異なる分類・ラベル・属性で記録される | REJECT |
| 数値として記録する値に NaN、Infinity、負値、契約外の 0 が混入し得る | REJECT |
| 成功、失敗、再試行、中断、早期終了など主要経路の観測結果が区別されている | OK |

## 保守性

| 基準 | 判定 |
|------|------|
| 関数/ファイルが複雑すぎる（追いにくい） | 警告 |
| 重複コードが多い | 警告 |
| 命名が不明確 | 修正が必要 |

## 技術的負債

| パターン | 判定 |
|---------|------|
| Issue番号・外部制約・除去条件のない TODO/FIXME | REJECT |
| Issue番号・外部制約・除去条件がある TODO/FIXME | 警告 |
| 空実装、スタブ、コメントアウトされた旧実装の放置 | REJECT |
| 理由なしの @ts-ignore, @ts-expect-error | 警告 |
| 理由なしの eslint-disable | 警告 |
| 非推奨APIの使用 | 警告 |

## 書き込み後の副作用チェック

ファイルやディレクトリへの書き込み後にスキャン・読み取りを行う処理がある場合、書き込んだファイルが意図せずスキャン対象に含まれていないか確認する。

| パターン | 例 | 判定 |
|---------|-----|------|
| 出力先ディレクトリのスキャン | facets をコピーした後に同ディレクトリを走査して構文を検出 | REJECT |
| 一時ファイルの読み戻し | 一時ディレクトリに書き込み後、同ディレクトリの全ファイルを処理 | REJECT |
| 自己参照的な処理 | 生成したファイルが次の処理パイプラインの入力になる | 警告 |

検証アプローチ:
1. ファイル書き込み後にディレクトリスキャン（readdir, glob 等）を行う箇所を確認
2. スキャン対象に書き込み先が含まれていないか確認
3. 含まれている場合、除外フィルタが存在するか確認
