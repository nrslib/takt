# 契約ライフサイクル知識

契約の正しさは端から端までの性質である。値や振る舞いの約束は、生成、検証、変換、保存、読取、導出、解消のすべての経路が同じ意味を保つときだけ有効になる。

## ライフサイクル網羅

変更した契約は宣言や主呼び出し元だけで終えず、影響する境界をすべて追跡する。

| 基準 | 判定 |
|------|------|
| 変更したフィールドや振る舞いを生成元または型定義だけ更新している | REJECT |
| 検証、直列化、導出値、別入口、利用元が旧来の意味を残している | REJECT |
| 影響するすべてのライフサイクル境界が同じ必須の意味を保つ | OK |
| 契約を受け取れず導出もできない根拠を示して、影響しない境界を除外している | OK |

## 等価経路

等価な入口と解消経路が、黙って異なる契約規則を適用してはならない。

| 基準 | 判定 |
|------|------|
| 一方の入口だけが制約を検証または保存し、等価な入口が迂回できる | REJECT |
| 再試行、再生、import、導出経路が仕様なしに契約の意味を変える | REJECT |
| 等価経路の意図的な違いを元要件または仕様が定めている | OK |

## 入口別経路と資源所有権

要件は公開入口と実行モードごとに、producer、validator、consumer が構成する別経路を持つ。資源の寿命は所有者、所有権移譲、last consumer で決まり、調査や再開に必要な永続 artifact は一時資源と異なる。

| 基準 | 判定 |
|------|------|
| CLI、API、pipeline、再実行などで producer、validator、consumer のいずれかが異なる | 別経路として扱う。1経路の充足だけでは他を証明しない |
| 到達不能または契約に非影響である根拠なく経路を除外する | REJECT |
| 所有者または所有権移譲が不明なまま、last consumer より前に資源を解放する | REJECT |
| 調査または再開に必要な永続 artifact を一時資源と同様に消去する | REJECT |
| 成功、失敗、中断、再試行で所有権と last consumer に沿った寿命契約を保つ | OK |

## 元契約に対する解消

finding の解消では、元の受入条件と同じ欠陥クラスの全経路を再確認する。

| 基準 | 判定 |
|------|------|
| 局所修正だけで、等価なライフサイクル経路を確認せず解消扱いにする | REJECT |
| 元要件と矛盾する現行実装の振る舞いを、テストが固定しているだけ | REJECT |
| 元要件が影響するすべての経路で成り立つ証跡がある | OK |
